# DOCX 작성 가이드

Coffeecong Books는 Word(DOCX) 파일을 변환 엔진(Mammoth.js)에 통과시켜 반응형 Ebook으로 만든다. **Word 스타일을 어떻게 쓰느냐가 변환 품질을 결정한다** — 이 가이드대로 작성하면 목차·페이지 분할·서식이 예상대로 나온다.

## 지원 파일

- **`.docx`만 지원** — 구버전 `.doc`나 암호화된 파일은 업로드 시 거부됨.
- 최대 20MB.

## 제목 스타일 = 목차 구조

Word의 **스타일 패널**에서 아래 스타일을 그대로 사용해야 한다. 폰트 크기·굵기만 손으로 바꾼 텍스트는 "제목"으로 인식되지 않는다.

| Word 스타일 (영문/한글 둘 다 인식) | 변환 결과 | 목차 노출 |
|---|---|---|
| `Title` / `제목` | 책 제목(H1) | 목차에는 안 뜸(책 자체 제목이므로) |
| `Heading 1` / `제목 1` | 장(chapter) 제목(H2) | ✅ 목차 표시 |
| `Heading 2` / `제목 2` | 절 제목(H3) | ✅ 목차 표시 |
| `Heading 3` / `제목 3` | 소제목(H4) | 목차에는 안 뜸(본문 내 강조용) |
| `Quote` / `Intense Quote` / `인용` / `강조 인용` | 인용구(blockquote) | — |

문서 안에 책 제목은 **한 번만** `Title`/`제목` 스타일로 넣는다. 장 구분은 `Heading 1`/`제목 1`, 그 아래 절은 `Heading 2`/`제목 2`로 — 목차 패널(Reader의 "목차로" 버튼)은 정확히 이 두 레벨만 보여준다.

## 지원되는 서식

- 일반 문단
- 굵게(bold) / 기울임(italic) / 밑줄(underline)
- 하이퍼링크
- 글머리기호 목록·번호 목록 (중첩 목록도 되지만, 페이지 경계에서 목록 중간을 자르지 않고 통째로 다음 페이지로 넘기므로 아주 긴 목록은 페이지 하나를 다 차지할 수 있음)
- 표 (기본적인 행/열 — 페이지보다 넓으면 가로 스크롤 처리됨, 표 중간을 페이지 경계에서 쪼개지는 않음)
- 이미지 (삽입된 이미지는 자동으로 WebP로 변환되어 저장되고, 원본 비율을 유지한 채 페이지에 맞게 리사이즈됨)

## 변환 시 제거되는 것

보안을 위해 다음은 통째로 제거되거나(내용까지) 태그만 벗겨진다 — Word 문서에 이런 게 있을 일은 거의 없지만, 다른 도구로 만든 `.docx`를 재사용할 경우 주의:

- `<script>`, `<iframe>`, `<object>`, `<embed>`, `<form>`, 버튼/입력 요소 — 내용째 제거
- 인라인 이벤트 핸들러(`onclick=` 등), `javascript:` 링크
- `data:` URL 중 이미지가 아닌 것
- 허용 목록에 없는 태그는 벗겨지되 **텍스트 내용은 보존**됨(예: 임의의 `<div>`는 사라지고 안의 글자만 남음)

## 잘 안 될 때

- **목차가 비어 있다** → `Heading 1`/`Heading 2` 스타일을 실제로 적용했는지 확인 (수동으로 폰트만 키운 텍스트는 인식 안 됨).
- **이미지가 안 보인다** → 삽입이 아니라 링크로만 걸어둔 이미지는 변환되지 않는다. Word에 실제로 삽입(첨부)된 이미지만 지원.
- **표/목록이 페이지를 넘어가며 어색하게 잘린다** → 현재 표/목록 중간 분할은 지원하지 않는 범위(스펙상 의도된 제한) — 아주 긴 표/목록은 여러 개로 나눠 작성하는 걸 권장.
- **미리보기와 저장 후 내용이 다르다** → 미리보기는 실제 저장 로직과 동일한 변환을 쓰므로 다르면 안 되는 게 정상 — 다르다면 버그이니 보고할 것.
